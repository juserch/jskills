# block-break — 能力边界

## 定位

- **分类**：hammer（行为约束，OUTPUT 是对 agent 的执行指令）
- **触发**：`/block-break` 显式命令 + UserPromptSubmit hook（matcher 捕获挫败语："又错了 / 别偷懒 / stop spinning / try harder"）
- **本质**：agent 卡住时强制 L0-L4 压力升级，穷尽一切方案前不允许"做不到"

## ✅ 能解决的问题

| 问题类型 | 机制 | 典型案例 |
|---------|------|---------|
| 过早放弃 | 三条红线 + 压力升级 | "这超出我的能力" 被 L2 拷问挡下 |
| 合理化"做不到" | 抗合理化表 14 条 | "可能是环境问题" 被识别为借口模式 |
| 反复试同一方案 | 五步方法论的"换新方案" | 第 3 次 `npm install` 失败 → L2 强制换路径 |
| 浅层诊断 | L3 的 7 项检查清单 | 没读源码 / 没搜完整错误 → 清单堵住 |
| 环境性假设过早 | "先本地重现" 规则 | "CI 才会挂" → 要求先本地跑 |

## ❌ 不能解决的问题

### 设计边界内（故意不管）

| 问题 | 为什么不管 |
|------|-----------|
| 方案本身的对错 | block-break 逼**穷尽**，不做**质量判断**。穷尽了一堆烂方案也算通过 |
| 错误问题框架 | 压力不修正"根因识别错"。方向错，压力让你把错方向走得更深 |
| 事实类断言准确性 | 那是 claim-ground 的职责 |
| 推理正确性 | 不做逻辑 / 数学 / 类型检查；压力只管"别停" |

### 超出设计能力（接不住）

| 问题 | 为什么接不住 |
|------|-------------|
| 用户真的应该停下 | 需求本身不合理（如用旧版本做只有新版本支持的事） |
| 硬性阻塞 | 真的没权限 / 硬件缺失 / 付费墙——L4 graceful exit 才触发，前面阶段会继续施压 |
| 情绪疲劳 | 用户累了想休息，block-break 还在 "try harder" |

## 🚫 不应使用场景

- **创作 / 发散思维任务**：压力扼杀发散，让 agent 收敛到"可交付"而非"有创意"
- **闲聊 / 娱乐**：触发了也无意义，徒增 verbose
- **用户明确要求暂停或接受现状**：继续施压会破坏信任

## 一句话定位

> block-break 是**穷尽式调试的发动机**——它保证 agent 不会过早退出，但不保证退出时的方案是最优、正确或应该执行的。它管"有没有努力"，不管"努力的方向对不对"。
