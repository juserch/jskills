# council-fuse — 能力边界

## 定位

- **分类**：crucible（多源融合，OUTPUT 是比输入更精炼的综合答案）
- **触发**：`/council-fuse <question>` 显式命令
- **本质**：3 个独立 agent（通才 / 批评者 / 专家）从不同视角回答 → 匿名评分 → Chairman 综合

## ✅ 能解决的问题

| 问题类型 | 机制 | 典型案例 |
|---------|------|---------|
| 单视角盲点 | 3 视角并行独立回答 | 只想到"该做"没想到"为什么不该" |
| 确认偏差 | Critic 视角对抗 | 批评者强制反驳通才方案 |
| 少数意见丢失 | 综合时保留有效异议 | 2:1 分歧时异议被记录不被消音 |
| 缺乏辩证过程 | 三视角独立打分 + Chairman 综合 | 决策有迹可循，不是"直觉选 A" |

## ❌ 不能解决的问题

### 设计边界内（故意不管）

| 问题 | 为什么不管 |
|------|-----------|
| 需要外部新数据 | 3 agents 只用训练知识思辨；不做 WebSearch |
| 时效敏感信息 | 没有 runtime 证据查询机制，看不到"此刻" |

### 超出设计能力（接不住）

| 问题 | 为什么接不住 |
|------|-------------|
| 共识错 | 3 agents 用同一训练基底，可能同步错；综合也是错的 |
| 高度主观问题 | "这个设计好看吗" 没有收敛点，3 视角各说各的 |
| 需要现场调查 | "用户 A 为什么点击率下降"——需要数据，不是辩论 |

## 🚫 不应使用场景

- **纯事实查询**：用 claim-ground，别用三视角辩论
- **创作性写作**：Chairman 综合会扼杀风格，让文本变"安全但无味"
- **速度优先的快速决策**：三 agent 顺序跑，开销不小

## 一句话定位

> council-fuse 是**基于训练知识的辩证引擎**——它让三个视角撞一撞，把单视角盲点暴露出来，但撞出来的结论仍受限于训练知识本身。
