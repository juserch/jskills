# claim-ground — 能力边界

## 定位

- **分类**：hammer（行为约束，事实类断言闸门）
- **触发**：description 自动识别 + UserPromptSubmit hook（matcher 捕获质疑语："真的吗 / are you sure / really?"）
- **本质**：把每个关于"此刻系统状态"的断言，绑定到可核验的 runtime 证据源

## ✅ 能解决的问题

| 问题类型 | 机制 | 典型案例 |
|---------|------|---------|
| 训练陈旧幻觉 | 规则 1: Runtime > Training | 系统 prompt 写了 Opus 4.7 但 agent 说 4.5 |
| 类推 / 惯例代替证据 | 规则 2: 先引用，后断言 | "通常都是这样" / "一般 React 18 还支持 X" |
| 举例当穷举 | 红线 2 + 规则 3 | help 里 `--model sonnet-4-6` → "只支持 sonnet" |
| 被反驳后换措辞 | 红线 3 + pushback hook | "对的我确认，最新就是 X"（没重查） |
| 作用域塌缩 | 规则 4（本地 vs 生态） | 用系统 prompt 的 GA latest 答全生态最新 |
| 不确定时胡编 | 规则 6 | 查不到 kubectl 就说"应该装了 kubernetes" |

## ❌ 不能解决的问题

### 设计边界内（故意不管）

| 问题 | 为什么不管 |
|------|-----------|
| 推理正确性 | 不是"引用问题"，是"思考问题"；应由 TDD / 类型检查兜 |
| 架构 / 战略判断 | 不是事实问题；应由 council-fuse 兜 |
| 多轮上下文漂移 | 事件驱动，不做定期 refresh；只在明显触发时激活 |

### 超出设计能力（接不住）

| 问题 | 为什么接不住 |
|------|-------------|
| 证据源本身错 | 保证"有源"，不保证"源对"。官方文档写错你引用它还是错 |
| 工具输出解读错 | `git log` 被 pager 截断你说"只有 3 次提交"——不管解读正确性 |
| 用户喂错证据 | 粘贴伪造"官方文档"→ 红线合规但事实错；不做可信度校验 |
| 静默的错前提 | 问题本身假设错（"为什么 3.9 比 3.10 慢"）——不挑战前提 |
| 未触发的隐式事实断言 | "用 React 19 新 hook 重构" 里隐藏假设抓不到 |

## 🚫 不应使用场景

- **纯推理任务**：解算法题、做架构设计——cite-then-conclude 无意义
- **创作 / 头脑风暴**：没有"事实待核"
- **解释训练知识**：如"async/await 是什么"——本就是训练范畴，不需要 runtime 证据

## 一句话定位

> claim-ground 是**事实类断言的入口闸门**——保证每个"此刻状态"事实有可验证来源，但管不了来源是否真实、是否被正确解读，也管不了非事实类判断。
