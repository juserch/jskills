## 工作原理

Council Fuse 将 Karpathy 的 LLM Council 模式浓缩为一条命令：

### 第一阶段：召集

三个代理**并行**启动，各自拥有不同视角：

| 代理 | 角色 | 模型 | 优势 |
|------|------|------|------|
| 通才 | 平衡、务实 | Sonnet | 主流最佳实践 |
| 批评者 | 对抗性、发现缺陷 | Opus | 边界情况、风险、盲点 |
| 专家 | 深入技术细节 | Sonnet | 实现精度 |

每个代理**独立**作答 — 它们无法看到彼此的回答。

### 第二阶段：评分

主席（主代理）将所有回答匿名化为回答 A/B/C，然后在 4 个维度上打分（0-10）：

- **正确性** — 事实准确性、逻辑严密性
- **完整性** — 各方面的覆盖程度
- **实用性** — 可操作性、现实适用性
- **清晰度** — 结构、可读性

### 第三阶段：综合

得分最高的回答成为骨架。其他回答中的独到见解被整合进来。批评者提出的合理异议作为注意事项保留。

## 使用场景

### 架构决策

```
/council-fuse Should we use microservices or a monolith for our 10-person team?
```

通才提供平衡的权衡分析，批评者质疑微服务的过度炒作，专家详述迁移模式。综合结果给出有条件的建议。

### 技术选型

```
/council-fuse Redis vs PostgreSQL for our job queue
```

三个不同角度确保您不会遗漏运维问题（批评者）、实现细节（专家）或务实的默认选择（通才）。

### 代码审查

```
/council-fuse Is this error handling pattern correct? <paste code>
```

一次获得主流验证、对抗性边界情况分析和深入技术验证。

## 输出结构

每次委员会审议产生：

1. **评分矩阵** — 对三个视角的透明评分
2. **共识分析** — 一致点和分歧点
3. **综合答案** — 融合后的最佳回答
4. **少数意见** — 值得关注的有效异议

---

## 自定义

### 更改视角

编辑 `agents/*.md` 来定义自定义委员会成员。可选的三人组合：

- 乐观者 / 悲观者 / 务实者
- 架构师 / 实现者 / 测试者
- 用户代言人 / 开发者 / 安全专家

### 更改模型

编辑每个代理文件中的 `model:` 字段：

- `model: haiku` — 经济实惠的委员会
- `model: opus` — 全重量级，用于关键决策

## 常见问题

**问：会消耗 3 倍的 token 吗？**
答：大致如此。三个独立回答加上综合。用于值得投入的决策。

**问：可以添加更多委员会成员吗？**
答：框架支持这样做 — 添加一个 `agents/*.md` 文件并更新 SKILL.md 工作流。不过，3 人是成本与多样性的最佳平衡点。

**问：如果某个代理失败了怎么办？**
答：主席将该成员评分为 0，并从其余回答中综合结果。优雅降级，不会崩溃。
