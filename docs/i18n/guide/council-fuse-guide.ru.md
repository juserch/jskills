## Как это работает

Council Fuse сводит паттерн LLM Council от Karpathy в одну команду:

### Этап 1: Созыв

Три агента запускаются **параллельно**, каждый с отличающейся точкой зрения:

| Агент | Роль | Модель | Сильная сторона |
|-------|------|--------|----------------|
| Универсал | Сбалансированный, практичный | Sonnet | Общепринятые лучшие практики |
| Критик | Оппонирующий, находит недостатки | Opus | Крайние случаи, риски, слепые зоны |
| Специалист | Глубокая техническая детализация | Sonnet | Точность реализации |

Каждый агент отвечает **независимо** — они не могут видеть ответы друг друга.

### Этап 2: Оценка

Председатель (основной агент) анонимизирует все ответы как Ответ A/B/C, затем оценивает каждый по 4 измерениям (0-10):

- **Корректность** — фактическая точность, логическая обоснованность
- **Полнота** — охват всех аспектов
- **Практичность** — применимость, пригодность в реальном мире
- **Ясность** — структура, читаемость

### Этап 3: Синтез

Ответ с наивысшей оценкой становится каркасом. Уникальные идеи из других ответов интегрируются. Обоснованные возражения критика сохраняются как оговорки.

## Примеры использования

### Архитектурные решения

```
/council-fuse Should we use microservices or a monolith for our 10-person team?
```

Универсал предоставляет сбалансированные компромиссы, критик ставит под сомнение хайп микросервисов, а специалист детализирует паттерны миграции. Синтез дает условную рекомендацию.

### Выбор технологий

```
/council-fuse Redis vs PostgreSQL for our job queue
```

Три разных угла зрения гарантируют, что вы не упустите операционные проблемы (критик), детали реализации (специалист) или прагматичный вариант по умолчанию (универсал).

### Ревью кода

```
/council-fuse Is this error handling pattern correct? <paste code>
```

Получите общепринятую валидацию, состязательный анализ крайних случаев и глубокую техническую верификацию за один проход.

## Структура вывода

Каждое обсуждение совета генерирует:

1. **Матрица оценок** — прозрачная оценка всех трёх перспектив
2. **Анализ консенсуса** — области согласия и разногласий
3. **Синтезированный ответ** — объединённый лучший ответ
4. **Мнение меньшинства** — обоснованные несогласные точки зрения, заслуживающие внимания

## Настройка

### Изменение перспектив

Отредактируйте `agents/*.md` для определения пользовательских членов совета. Альтернативные триады:

- Оптимист / Пессимист / Прагматик
- Архитектор / Реализатор / Тестировщик
- Защитник пользователя / Разработчик / Эксперт по безопасности

### Изменение моделей

Отредактируйте поле `model:` в каждом файле агента:

- `model: haiku` — экономичные советы
- `model: opus` — тяжеловесный состав для критических решений

## Когда использовать / Когда НЕ использовать

### ✅ Используйте когда

- Architectural decisions needing multiple perspectives
- You suspect confirmation bias in your current answer
- You want dissenting views preserved, not erased

### ❌ Не используйте когда

- Pure factual queries (use `claim-ground` instead)
- Creative writing (synthesis flattens style)
- Speed-critical decisions (3 agents run sequentially — not fast)

> Движок дебатов на базе тренировочных знаний — выявляет слепые зоны одной перспективы, но выводы всё ещё ограничены.

## Часто задаваемые вопросы

**В: Это стоит в 3 раза больше токенов?**
О: Да, приблизительно. Три независимых ответа плюс синтез. Используйте для решений, оправдывающих затраты.

**В: Можно ли добавить больше членов совета?**
О: Фреймворк поддерживает это — добавьте ещё один файл `agents/*.md` и обновите рабочий процесс в SKILL.md. Однако 3 — оптимальный баланс между стоимостью и разнообразием.

**В: Что произойдёт, если агент выйдет из строя?**
О: Председатель выставляет этому члену оценку 0 и синтезирует из оставшихся ответов. Плавная деградация, без сбоев.
