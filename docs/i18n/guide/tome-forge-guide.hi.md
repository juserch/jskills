## कमांड

| कमांड | यह क्या करता है | कब उपयोग करें |
|-------|----------------|---------------|
| `/tome-forge init` | ज्ञान आधार आरंभ करें | किसी भी निर्देशिका में नया KB शुरू करते समय |
| `/tome-forge capture [text]` | नोट, लिंक, या क्लिपबोर्ड की त्वरित कैप्चर | विचार नोट करना, URL सहेजना, क्लिपिंग |
| `/tome-forge capture clip` | सिस्टम क्लिपबोर्ड से कैप्चर करें | कॉपी की गई सामग्री को त्वरित सहेजना |
| `/tome-forge ingest <path>` | कच्चे माल को विकी में संकलित करें | `raw/` में पेपर, लेख, या नोट्स जोड़ने के बाद |
| `/tome-forge ingest <path> --dry-run` | बिना लिखे रूटिंग का पूर्वावलोकन | परिवर्तन करने से पहले सत्यापित करें |
| `/tome-forge query <question>` | विकी से खोजें और संश्लेषित करें | अपने ज्ञान आधार में उत्तर खोजना |
| `/tome-forge lint` | विकी संरचना की स्वास्थ्य जांच | कमिट से पहले, नियमित रखरखाव |
| `/tome-forge compile` | सभी नए कच्चे माल को बैच में संकलित करें | कई कच्ची फाइलें जोड़ने के बाद पकड़ना |

## यह कैसे काम करता है

Karpathy के LLM विकी पैटर्न पर आधारित:

```
raw materials + LLM compilation = structured Markdown wiki
```

### दो-स्तरीय वास्तुकला

| स्तर | स्वामी | उद्देश्य |
|------|--------|----------|
| `raw/` | आप | अपरिवर्तनीय स्रोत सामग्री — पेपर, लेख, नोट्स, लिंक |
| `wiki/` | LLM | संकलित, संरचित, क्रॉस-रेफरेंस्ड Markdown पृष्ठ |

LLM आपके कच्चे माल को पढ़ता है और उन्हें अच्छी तरह से संरचित विकी पृष्ठों में संकलित करता है। आप कभी `wiki/` को सीधे संपादित नहीं करते — आप कच्चे माल जोड़ते हैं और LLM को विकी बनाए रखने देते हैं।

### पवित्र खंड

हर विकी पृष्ठ में एक `## मेरी समझ का अंतर` खंड होता है। यह **आपका** है — LLM इसे कभी संशोधित नहीं करेगा। यहां अपनी व्यक्तिगत अंतर्दृष्टि, असहमतियां, या अंतर्ज्ञान लिखें। यह सभी पुनर्संकलनों में बना रहता है।

## KB खोज — मेरा डेटा कहां जाता है?

आप **किसी भी निर्देशिका** से `/tome-forge` चला सकते हैं। यह स्वचालित रूप से सही KB ढूंढता है:

| स्थिति | क्या होता है |
|--------|-------------|
| वर्तमान निर्देशिका (या मूल) में `.tome-forge.json` है | उस KB का उपयोग करता है |
| ऊपर की ओर कोई `.tome-forge.json` नहीं मिला | डिफ़ॉल्ट `~/.tome-forge/` का उपयोग करता है (आवश्यकता होने पर स्वचालित रूप से बनाया जाता है) |

इसका मतलब है कि आप पहले `cd` किए बिना किसी भी प्रोजेक्ट से नोट्स कैप्चर कर सकते हैं — सब कुछ आपके एकल डिफ़ॉल्ट KB में जाता है।

प्रति प्रोजेक्ट अलग KB चाहिए? उस प्रोजेक्ट निर्देशिका के अंदर `init .` का उपयोग करें।

## कार्यप्रवाह

### 1. आरंभ करें

```
/tome-forge init                  # Create default KB at ~/.tome-forge/
/tome-forge init .                # Create KB in current directory
/tome-forge init ~/research-kb    # Create KB at a specific path
```

आरंभ करने के बाद, KB निर्देशिका इस तरह दिखती है:

```
~/.tome-forge/               # (or wherever you initialized)
├── .tome-forge.json         # KB marker (auto-generated)
├── CLAUDE.md                # KB schema and rules
├── index.md                 # Wiki page index
├── .gitignore
├── logs/                    # Operation logs (monthly rotation)
│   └── 2026-04.md           # One file per month, never grows too large
├── raw/                     # Your source materials (immutable)
└── wiki/                    # LLM-compiled wiki (auto-maintained)
```

### 2. कैप्चर करें

**किसी भी निर्देशिका** से, बस चलाएं:

```
/tome-forge capture "attention is fundamentally a soft dictionary lookup"
/tome-forge capture https://arxiv.org/abs/1706.03762
```

त्वरित कैप्चर `raw/captures/{date}/` में जाते हैं। लंबी सामग्री के लिए, फाइलें सीधे `raw/papers/`, `raw/articles/` आदि में डालें।

### 3. अंतर्ग्रहण

```
/tome-forge ingest raw/papers/attention-is-all-you-need.md
```

LLM कच्ची फाइल पढ़ता है, इसे सही विकी पृष्ठ(ओं) पर रूट करता है, और आपके व्यक्तिगत नोट्स को संरक्षित करते हुए नई जानकारी को मर्ज करता है।

### 4. क्वेरी करें

```
/tome-forge query "what is the relationship between attention and transformers?"
```

आपके विकी से उत्तर संश्लेषित करता है, विशिष्ट पृष्ठों का हवाला देते हुए। आपको बताता है कि जानकारी गायब है और कौन सा कच्चा माल जोड़ना चाहिए।

### 5. रखरखाव

```
/tome-forge lint
/tome-forge compile
```

Lint संरचनात्मक अखंडता की जांच करता है। Compile पिछले संकलन के बाद से सब कुछ नया बैच में अंतर्ग्रहण करता है।

## विकी पृष्ठ प्रारूप

हर विकी पृष्ठ एक सख्त टेम्पलेट का पालन करता है:

```yaml
---
domain: ai
maturity: growing        # draft | growing | stable | deprecated
last_compiled: 2026-04-15
source_refs:
  - raw/papers/attention.md
confidence: medium       # low | medium | high
compiled_by: claude-opus-4-6
---
```

आवश्यक खंड:
- **मूल अवधारणा** — LLM-रखरखाव ज्ञान
- **मेरी समझ का अंतर** — आपकी व्यक्तिगत अंतर्दृष्टि (LLM द्वारा कभी नहीं छुआ जाता)
- **खुले प्रश्न** — अनुत्तरित प्रश्न
- **संबंध** — संबंधित विकी पृष्ठों के लिंक

## अनुशंसित ताल

| आवृत्ति | कार्रवाई | समय |
|---------|---------|------|
| **दैनिक** | विचारों, लिंक, क्लिपबोर्ड के लिए `capture` | 2 मिनट |
| **साप्ताहिक** | सप्ताह के कच्चे माल को बैच-प्रोसेस करने के लिए `compile` | 15-30 मिनट |
| **मासिक** | `lint` + मेरी समझ का अंतर खंडों की समीक्षा | 30 मिनट |

**वास्तविक समय अंतर्ग्रहण से बचें।** बार-बार एकल-फाइल अंतर्ग्रहण विकी की सुसंगतता को खंडित करता है। साप्ताहिक बैच संकलन बेहतर क्रॉस-रेफरेंस और अधिक सुसंगत पृष्ठ उत्पन्न करता है।

## कब उपयोग करें / कब उपयोग न करें

### ✅ इन मामलों में उपयोग करें

- Building a personal knowledge base from scattered notes
- You want LLM-compiled wiki that preserves human insights
- Zero-infra solution (just Markdown + Git)

### ❌ इन मामलों में उपयोग न करें

- Team collaboration or real-time sync
- Ad-hoc notes (too structured — plain Markdown is fine)
- Transactional data (use a real database)

> LLM-compiled व्यक्तिगत library — human insights सुरक्षित रखता है, केवल व्यक्ति के लिए; team sync या permission नहीं।

## FAQ

**प्रश्न: विकी कितना बड़ा हो सकता है?**
उत्तर: 50 पृष्ठों से कम पर, LLM सब कुछ पढ़ता है। 50-200 पृष्ठों पर, यह नेविगेट करने के लिए इंडेक्स का उपयोग करता है। 200 से अधिक पर, डोमेन शार्डिंग पर विचार करें।

**प्रश्न: क्या मैं विकी पृष्ठों को सीधे संपादित कर सकता हूं?**
उत्तर: केवल `## मेरी समझ का अंतर` खंड। बाकी सब कुछ अगले अंतर्ग्रहण/संकलन पर अधिलेखित हो जाएगा।

**प्रश्न: क्या इसे वेक्टर डेटाबेस की आवश्यकता है?**
उत्तर: नहीं। विकी सादा Markdown है। LLM सीधे फाइलें पढ़ता है — कोई एम्बेडिंग नहीं, कोई RAG नहीं, कोई इंफ्रा नहीं।

**प्रश्न: मैं अपने KB का बैकअप कैसे लूं?**
उत्तर: यह सब एक git रिपॉजिटरी में फाइलें हैं। `git push` और हो गया।

**प्रश्न: अगर LLM विकी में गलती करता है तो क्या?**
उत्तर: `raw/` में सुधार जोड़ें और पुनः अंतर्ग्रहण करें। मर्ज एल्गोरिथम अधिक प्रामाणिक स्रोतों को प्राथमिकता देता है। या अपनी मेरी समझ का अंतर में असहमतियां नोट करें।
