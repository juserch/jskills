## 仕組み

Council Fuse は Karpathy の LLM Council パターンを1つのコマンドに凝縮します：

### ステージ 1: 召集

3つのエージェントがそれぞれ異なる視点を持って**並列で**生成されます：

| エージェント | 役割 | モデル | 強み |
|--------------|------|--------|------|
| ゼネラリスト | バランス型、実践的 | Sonnet | 主流のベストプラクティス |
| 批評家 | 対立的、欠陥を発見 | Opus | エッジケース、リスク、盲点 |
| スペシャリスト | 深い技術的詳細 | Sonnet | 実装の精度 |

各エージェントは**独立して**回答します — 互いの回答を見ることはできません。

### ステージ 2: 採点

議長（メインエージェント）がすべての回答を回答 A/B/C として匿名化し、4つの次元で採点します（0-10）：

- **正確性** — 事実の正確さ、論理的妥当性
- **完全性** — すべての側面のカバー率
- **実用性** — 実行可能性、現実世界での適用性
- **明瞭性** — 構造、可読性

### ステージ 3: 統合

最高得点の回答が骨格となります。他の回答からのユニークな洞察が統合されます。批評家の妥当な異議は注意事項として保持されます。

## ユースケース

### アーキテクチャの意思決定

```
/council-fuse Should we use microservices or a monolith for our 10-person team?
```

ゼネラリストがバランスの取れたトレードオフを提供し、批評家がマイクロサービスの誇大宣伝に異議を唱え、スペシャリストが移行パターンを詳述します。統合結果は条件付きの推奨を提供します。

### 技術選定

```
/council-fuse Redis vs PostgreSQL for our job queue
```

3つの異なる角度により、運用上の懸念（批評家）、実装の詳細（スペシャリスト）、実用的なデフォルト（ゼネラリスト）を見逃しません。

### コードレビュー

```
/council-fuse Is this error handling pattern correct? <paste code>
```

主流の検証、対立的なエッジケース分析、深い技術的検証を一度に取得できます。

## 出力構造

すべての委員会審議は以下を生成します：

1. **採点マトリックス** — 3つの視点の透明な採点
2. **合意分析** — 一致点と不一致点
3. **統合回答** — 融合されたベストアンサー
4. **少数意見** — 注目に値する有効な反対意見

## カスタマイズ

### 視点の変更

`agents/*.md` を編集してカスタム委員会メンバーを定義します。代替の3人組：

- 楽観主義者 / 悲観主義者 / 現実主義者
- アーキテクト / 実装者 / テスター
- ユーザー擁護者 / 開発者 / セキュリティ専門家

### モデルの変更

各エージェントファイルの `model:` フィールドを編集します：

- `model: haiku` — コスト効率の良い委員会
- `model: opus` — 重要な意思決定のための全ヘビー級

## FAQ

**Q: トークン消費は3倍になりますか？**
A: おおよそそうです。3つの独立した回答と統合で構成されます。投資に値する意思決定に使用してください。

**Q: 委員会メンバーを追加できますか？**
A: フレームワークはサポートしています — `agents/*.md` ファイルを追加し、SKILL.md のワークフローを更新してください。ただし、コスト対多様性のバランスでは3人が最適です。

**Q: エージェントが1つ失敗した場合はどうなりますか？**
A: 議長はそのメンバーを0点とし、残りの回答から統合します。グレースフルデグラデーションにより、クラッシュは発生しません。
